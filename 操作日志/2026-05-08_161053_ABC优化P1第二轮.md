# P1 第二轮优化（CF Insights / GTM Clarity / next.config）

- 日期：2026-05-08 16:10

## A. CF 关 Cloudflare Insights ✅

路径：dash.cloudflare.com → 数据分析 → Web Analytics → 站点 aierxuanlaptop.com → 管理站点 → RUM 选「禁用」→ 更新

验证：`curl https://www.aierxuanlaptop.com/en | grep cloudflareinsights` 返回 0 个匹配。
节省：11 KiB beacon.min.js + 减少 main thread 阻塞。

## B. GTM Clarity 延迟加载 — 主动跳过

权衡：
- 收益仅 25 KiB，且 Clarity 模板触发器不能直接改 Window Loaded
- 完整改造需 4-5 步 GUI 操作 + 创建新触发器 + 工作区提交
- 性价比远不如 C，故放弃

保留 Clarity 不改：现有 All Pages 触发，对 PSI 有微小影响但 Clarity session recording 数据照常收集。

## C. next.config.mjs 纳入仓库 + 优化 ✅

`commit 196f072 chore(next): 把 next.config.mjs 纳入仓库 + 启用 optimizePackageImports`

变更：
- 把 VPS 上未追踪的 next.config.mjs 纳入 git（之前每次部署都是 VPS 独有）
- 移除废弃的 `eslint.ignoreDuringBuilds`（Next.js 16 不再支持，build 会报警告）
- 加 `experimental.optimizePackageImports` 给 lucide-react / @heroicons/react

实际收益：
- 本地 build 28s → 17s（移除 eslint plugin 加载）
- chunks 大小未明显减（Next.js 16 + Turbopack 默认已优化得相当好）
- 437 KiB unused JS 主要是 React/Next 框架代码，无法 split

## 真相：C 的 PSI 改善预期小

unused JS 437 KiB 的大头是不可避免的 framework code。Next.js 16 + Turbopack 默认就是路由级 split，没有更多空间。

唯一真能让 PSI Mobile Performance 上 80+ 的剩余路径：
- 删 Microsoft Clarity（25 KiB，但失去录屏数据）
- 改 SSR 策略让首页变 ISR static（需要重构 root layout 不读 cookie）
- 减少 GTM/Yandex 两个 tag 之一

这些都是更深的改造，今晚不适合做。
